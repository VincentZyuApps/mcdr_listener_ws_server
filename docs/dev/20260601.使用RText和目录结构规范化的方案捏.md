## 使用 RText 和目录结构规范化的方案

这份文档用于规划 `mcdr_listener_ws_server` 的一次偏工程化整理，目标不是一次性大重写，而是在尽量不破坏现有功能的前提下，把插件逐步整理到更接近 MCDR 社区习惯、也更方便后续发版和维护的状态。

当前插件已经能工作，但存在几个比较明显的问题：

- 玩家可见消息里混用了 `str`、`§` 颜色码、`tellraw` SNBT 字符串。
- 入口文件 `__init__.py` 承担了过多职责，事件监听、命令注册、状态管理都堆在一起。
- `ws_server.py` 同时负责 WebSocket 生命周期、协议解析、Minecraft 消息格式化、图片消息广播，职责偏重。
- `image_handler.py` 既做下载缓存、又做消息组件拼接、又做实体生成，后续维护成本高。
- 项目目录比较扁平，缺少 `lang/`、更明确的模块分层和开发文档组织。
- 有一些示例代码和临时文件仍然留在正式插件目录里，例如 `qwq_lib.py`、`tmp.md`。

注：随着方案推进，上面其中一部分问题已经开始被解决，例如 `qwq_lib.py` / `tmp.md` 已移除、`config.py` / `lang/` 已初步引入。这里保留该段是为了记录原始问题背景，具体完成状态请以下方“当前完成度与缺口”为准。

结合 `Games_AI` 和 `UnifiedHandler` 两个仓库，可以借鉴的不是“照搬目录”，而是它们体现出来的几条原则：

- 顶层目录干净，插件主体代码尽量集中在同名包目录内。
- 配置、核心逻辑、加载入口、资源文件分开。
- 插件元数据、README、语言资源、测试和开发文档彼此职责明确。
- 文本输出尽量遵循 MCDR 风格，优先使用 `RText` / `RTextList` / 翻译文本，而不是直接硬拼原版样式码。

---

## 改造目标

这次规范化建议分成 4 个目标：

1. 统一玩家可见文本输出风格，减少 `§` 颜色码和纯字符串直出。
2. 重新梳理目录结构，让“入口层 / 协议层 / 展示层 / 工具层”职责清晰。
3. 为后续发版做准备，靠近 MCDR 插件常见组织方式。
4. 保持功能兼容，避免一次性推翻现有逻辑。

## 0、从审核视角看“理想标准 MCDR 插件”还需要什么

仅仅“功能能跑”并不等于“审核友好”。结合 `UnifiedHandler`、`Games_AI` 这类结构更规整的插件，以及 MCDR 作者更偏好的风格，一个更理想、更标准、更容易通过人工审核的插件，通常还会具备这些特征：

- 顶层目录干净，没有明显临时文件、示例草稿文件、运行产物目录污染仓库。
- 入口清晰，`__init__.py` 尽量只负责装配，不堆大量业务逻辑。
- 玩家可见消息尽量使用 `RText`、`RTextList`、翻译文本，而不是大量 `§` 颜色码。
- 具备基础国际化能力，至少有 `lang/` 和一组核心翻译键。
- 具备最小配置能力，而不是把 host、port、缓存路径、展示尺寸全部硬编码。
- 有最小测试骨架，至少覆盖几个核心纯函数。
- README、插件元数据、目录结构、配置说明彼此一致，不互相打架。
- 仓库中不保留明显不该提交的内容，例如 `__pycache__/`、运行日志目录、临时 md、示例脚本。

换句话说，一个“理想标准 MCDR 插件”不一定很大，但至少应该满足：

1. 功能边界清楚
2. 面向玩家的文本规范
3. 仓库形态整洁
4. 配置和文档可交付
5. 对维护者之外的人也可读

## 一、RText 使用规范

### 1.1 哪些地方应该优先改成 RText

优先级最高的是所有“面向命令源和玩家的文本”：

- `source.reply(...)`
- `server.tell(...)`
- `server.say(...)` 中的普通聊天广播
- `register_help_message(...)`

这些地方目前如果只是简单文字提示，不应该再继续使用：

- `§c此命令只能由玩家执行`
- `§` 开头的 MOTD 样式字符串
- 纯字符串硬编码帮助消息

更合适的做法是：

- 错误提示用 `RText(..., RColor.red)`
- 普通说明用 `RText(..., RColor.gray)` 或默认颜色
- 结构化聊天用 `RTextList(...)`
- 命令建议、点击事件、悬浮提示用 `RText` 的 click / hover API

### 1.2 哪些地方不适合强行“纯 RText 化”

不是所有地方都应该强行改成“只剩 RText，不再碰原版命令”。

对于这个插件，更准确的目标应该是：

- 玩家可见聊天消息尽量统一使用 `RText` / `RTextList`
- 帮助消息、命令错误提示、状态提示统一使用翻译文本 + `RText`
- 必须依赖 Minecraft 原生命令执行的展示层，继续保留 `execute` / `tellraw` / `summon text_display`
- 但在能控制的地方，尽量改成“先构造 `RText`，再序列化成 JSON 文本组件”

以下场景继续保留原版命令层是合理的：

- 需要 `summon minecraft:text_display`
- 需要写实体 NBT
- 需要 `execute as/at/anchored` 这种命令上下文
- 需要发出一整条原版 `tellraw` 命令

所以本插件里：

- `image_handler.py` 中和 `text_display` 召唤、清理相关的逻辑，本质上不可能只靠 `RText` 完成
- 图片消息里的“可点击入口文本”其实可以由 `RText` 生成，再转换成 `tellraw` 需要的 JSON
- 简单加载提示、错误提示、完成提示应该优先切换到 `server.tell(player, RText(...))` 或翻译文本

### 1.3 RText 改造建议顺序

建议按这个顺序做：

1. 先把 `source.reply('§c...')` 改掉。
2. 再把 `register_help_message()` 改成 `RText` 或翻译文本。
3. 再把 `ws_server.py` 中普通群聊广播的 `§` 格式字符串改成 `RTextList`。
4. 最后处理 `image_handler.py` 中那些只是简单文本反馈的提示。
5. 在图片消息链路里，把“手写 SNBT 文本组件”逐步改成“`RText` 组件 -> JSON 序列化 -> `tellraw`”。

### 1.4 结合当前代码的明确判断

对照当前实现，结论可以直接写清楚：

- `message_formatter.py` 已经在走正确路线，普通平台聊天广播使用 `RTextList` 是符合预期的
- `__init__.py` / `commands.py` / `events.py` 里的命令回复、帮助文本、状态提示，应继续优先走 `server.reply` / `server.tell` + 翻译文本
- `ws_server.py` 在“无图片消息”的分支里，继续使用 `server.say(RTextList(...))` 是最理想的
- `ws_server.py` 在“带图片消息”的分支里，目前仍然手写 `tellraw @a [...]`，这一段是后续最值得继续 RText 化的地方
- `image_message.py` 里当前 `create_clickable_image_text()` 还是手写 SNBT，这不算最理想；更理想的是先构造 `RText('[图片#x]')`，再挂 click / hover 事件，再序列化
- `image_handler.py` 里的 `text_display` 实体召唤、定时清理逻辑，仍然应该保留为原版命令层，不建议为了“纯 RText”去硬改

## 二、目录结构规范化目标

原始目录结构：

```text
mcdr_listener_ws_server/
  README.md
  requirements.txt
  mcdreforged.plugin.json
  docs/
  logs/
  mcdr_listener_ws_server/
    __init__.py
    ws_server.py
    image_handler.py
    log_utils.py
    qwq_lib.py
    tmp.md
```

这个结构能跑，但不够清晰。建议逐步整理为：

```text
mcdr_listener_ws_server/
  mcdreforged.plugin.json
  README.md
  requirements.txt
  lang/
  docs/
    dev/
  tests/
  mcdr_listener_ws_server/
    __init__.py
    config.py
    commands.py
    events.py
    ws_server.py
    message_formatter.py
    image_handler.py
    image_cache.py
    image_message.py
    logger.py
    utils.py
```

这里不是要求一步到位，而是提供拆分方向。

### 2.1 推荐拆分说明

`__init__.py`

- 只保留插件入口
- 只做加载、卸载、注册命令、注册事件监听、初始化全局对象
- 避免继续承载具体业务逻辑

`commands.py`

- 放 `!!view_image` 等命令注册与处理逻辑
- 所有 `CommandSource` 交互放到这里
- 方便统一处理权限、错误提示、RText 回复

`events.py`

- 放 `on_player_joined`、`on_player_left`、`on_user_info` 等事件监听逻辑
- 把“监听事件”和“如何广播到 WS”解耦

`message_formatter.py`

- 专门负责把平台消息转换成 Minecraft 侧显示文本
- 包括：
  - 普通文本格式化
  - RTextList 构造
  - 普通消息前缀拼接

`ws_server.py`

- 只保留 WebSocket 服务启动、连接管理、消息分发
- 不负责太多 Minecraft 展示细节

`image_handler.py`

- 保留图片下载、缓存、转换、展示
- 把“消息文本组件生成”慢慢移出到独立 helper

`image_cache.py`

- 负责图片下载和缓存
- 避免 `image_handler.py` 同时承担 IO 缓存职责

`image_message.py`

- 负责图片占位符替换
- 负责图片入口文本组件拼接
- 后续适合作为“手写 SNBT -> RText 序列化”改造的主要落点

`config.py`

- 统一放插件配置模型与加载逻辑
- 当前已使用 MCDR 常见的 `config/<plugin_id>/config.json` 路径
- host、port、缓存目录、图片尺寸上限、显示时长等默认值也已收敛到这里

## 三、建议引入的资源和辅助目录

### 3.1 `lang/`

如果准备更规范地适配 MCDR 生态，建议尽早加语言资源目录。

最初不一定要铺很多翻译键，但至少建议把这些文案纳入翻译：

- 命令错误提示
- 图片加载中 / 下载失败 / 处理失败 / 已展示
- WebSocket 服务启动成功 / 失败
- 帮助消息

这样后续：

- `register_help_message()` 可以直接用翻译文本
- `source.reply()` 可以根据玩家语言自动显示
- 发版时观感会更像标准 MCDR 插件

### 3.2 `tests/`

建议至少补最便宜的测试：

- 文本格式化函数测试
- 图片标记替换函数测试
- 文本清理函数测试

不需要一开始就对 WebSocket 服务做全套集成测试，但纯函数测试很值得加。

### 3.3 `docs/dev/`

你现在已经开始建 `docs/dev/`，这是对的。

后续建议把开发文档分成几类：

- 架构说明
- 协议说明
- 发版 checklist
- RText 使用规范
- 图片消息实现说明

### 3.4 `resources/` 或其他运行期资源目录

如果后续插件需要：

- 默认配置模板
- 协议示例 JSON
- 静态示例资源
- 图片或文本文档资源

建议专门建资源目录，而不是继续散落在代码目录或 README 里。

目前这个插件不一定立刻需要，但从参考项目的组织方式来看，预留这个思路是有价值的。

目标：减少 `ws_server.py` 的职责。

## 九、不建议现在做的事

以下事情先别一起上：

- 一次性重写整个图片展示系统
- 一次性把所有 `tellraw` / `execute` / `text_display` 全部替换成 RText
- 一次性上很复杂的配置系统
- 一次性补很重的集成测试

原因很简单：收益不如前面几项高，而且容易把现有能跑的功能打散。

## 十、结论

这次规范化最核心的方向不是“把代码写得很花”，而是：

- 把玩家可见文本尽量收敛到 MCDR 推荐的 `RText` 路线
- 对必须保留的原版命令层，尽量改成由 `RText` 序列化生成聊天组件
- 把入口文件减负
- 把 WebSocket、消息格式化、图片处理三块职责拆开
- 给项目补上更标准的目录与文档骨架

建议采用“先低风险拆分，再逐步 RText 化，再加 `config.py` / `lang/` / `tests/` / 仓库清理”的顺序推进。这样每一步都能提交、能验证、能回滚，也最适合你这个已经在使用中的插件。
