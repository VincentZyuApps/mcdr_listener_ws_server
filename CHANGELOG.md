# Changelog

> 基于仓库实际 `git log`、`git diff` 与 `mcdreforged.plugin.json` 的历史版本字段整理。  
> 版本划分优先以 `plugin.json` 中真实出现过的 `version` 为准；未改动版本号但有独立提交的内容，会在相邻版本中单独注明。  
> 本项目在近几个版本中，除了功能迭代，也持续推进了 **从“直接拼接原版字符串/命令”到“更多使用 MCDR API、RText、翻译文本、模块化结构”** 的风格演进，这里也一并记录。 

## [0.6.3-beta.2] - 2026-06-02

### 🎉 版本概览
- 🚀 当前最新版本推进至 `0.6.3-beta.2`，重点放在配置反序列化兜底与安装文档补强。
- 🛡️ 配置加载开始考虑“旧配置 / 手改配置 / 字段结构升级”下的兼容失败场景，启动容错更稳。
- 📖 README 安装流程补上 GitHub / Gitee `git clone` 指令，首次部署路径更清楚。

### 🔧 Changed
- ⚙️ `config.py` 不再直接 `target_class=PluginConfig` 反序列化，而是先读取原始配置，再手动 `PluginConfig.deserialize(raw)`。
- 🛡️ 当反序列化失败时，会记录 warning 并回退到默认 `PluginConfig()`，降低配置结构升级后直接启动失败的风险。
- 📝 中英文 README 的 Installation 段落补充 GitHub / Gitee 克隆命令，减少新用户手动找仓库地址的步骤。

### 💡 阶段观察
- 🪜 这一版不属于“大功能”版本，更像是为前一个 `0.6.3-beta.1` 的新配置项与文档重构做收尾。
- ✅ 对维护者来说，这种“补兜底 + 补文档入口”的提交很关键，能降低配置升级时的炸服概率。

## [0.6.2-beta.1] - 2026-06-02

### 🎉 版本概览

### 💡 阶段观察
- ⚖️ 这是一个“内部实现重构幅度大于表面功能增量”的版本。
- 🚀 对图片渲染链路来说，固定位置模式是一次明显的工程化提升，尤其有利于后续继续优化性能和显示稳定性。

## [0.4.0-beta.5+20260601] - 2026-06-01 ~ 2026-06-02

### 🎉 版本概览
- 🚀 插件进入 `0.4.x` 阶段，完成一次明显的工程化整理。
- 🧩 代码结构从偏单体实现，升级为按职责拆分的模块化组织。
- 🌐 文档体系同步升级为中英文双语，并补充了预览图、外部框架链接和扩展接入说明。
- 🛠️ 更重要的是，这一阶段开始更系统地向 **MCDR 官方 API 风格** 靠拢，尤其体现在 `RText`、`RTextList`、翻译文本与帮助消息注册的使用上。

### 🔨 Refactor
- 🗂️ 拆分命令、配置、事件、图片处理、消息格式化、翻译等模块，项目结构更清晰。
- 🧱 新增 `config`、`lang`、`docs`、`CHANGELOG` 等组成部分，仓库布局更完整。
- 🧹 移除废弃文件 `qwq_lib.py`、`tmp.md`，减少历史遗留负担。
- 🖼️ 保留当前运行环境下可用的图片点击兼容实现，说明图片渲染链路已经过一次兼容性筛选。
- 🏷️ 将 WebSocket 事件类型 `player_msg` 统一重命名为 `player_chat`，与 README 表述保持一致，语义更准确。
- ⚙️ `__init__.py` 中原本较重的启动/注册逻辑被拆分到 `commands.py`、`events.py`、`config.py`、`translator.py` 等模块，插件主入口更接近“装配层”而不是“所有逻辑都堆在一起”。

### 🧭 RText / MCDR API 风格演进
- 🎨 早期群聊转发主要依赖 `§` 样式码直接拼接字符串；这一阶段新增了独立的 `message_formatter.py`，把普通平台聊天消息封装为 `RTextList(...)`。
- ✨ `format_platform_message()` 现已使用 `RText` + `RColor` + `RStyle` 组合出群名、群号、昵称、正文等结构化片段，而不是继续手写 `§6§l...§r` 这种旧式格式串。
- 📣 `ws_server.py` 在“无图片消息”的普通广播分支中，已经改为 `server.say(RTextList(...))` 的方向，这属于比较标准的 MCDR 玩家可见消息输出方式。
- 🈯 新增 `translator.py`，通过 `server.rtr(...)` 封装翻译文本调用，使帮助消息、命令提示、错误提示不再局限于硬编码中文字符串。
- 🌍 新增 `lang/zh_cn.yml` 与 `lang/en_us.yml`，开始具备真正的插件内语言资源结构，而不只是 README 层面的双语说明。
- 🧾 `commands.py` 中的 `register_help_message('!!view_image <url>', tr(...))` 体现了对 MCDR 帮助系统的接入，命令说明不再只是散落在 README 里。
- 🚫 命令错误提示从早期常见的 `source.reply('§c...')` 风格，逐步转向 `reply_tr(...)` 和翻译文本回复，更符合 MCDR 插件生态中的可维护写法。
- 🧱 `image_handler.py` 中图片白名单拦截消息已经使用 `RTextList` 构造，`Blocked` 标记、前缀说明、被拦截的 host 都能分段着色展示。
- 🧼 文本进入 `RText` 之前，还会先经过 `sanitize_for_console_encoding()` 等清洗流程，降低控制字符、平台编码问题、异常样式字符对控制台和聊天输出的干扰。

### 🖼️ 关于“为什么不是所有东西都纯 RText 化”
- ⚖️ 这一阶段并不是“所有消息 100% 只靠 RText”，而是进入了更现实的 **混合架构**。
- 💬 玩家可见的普通文字消息、帮助消息、错误提示、状态提示，已经明显在向 `RText` / `RTextList` / 翻译文本收敛。
- 🧷 但涉及图片点击入口、`tellraw` 组合消息、`text_display` 实体渲染、定时清理等内容时，仍然必须保留一部分原版命令层逻辑。
- 🧠 这不是“没改完”，而是因为图片展示本质上依赖 Minecraft 原生命令和显示实体系统，不能简单用 `RText` 一把替代。
- 🔩 当前实现已经比早期更进一步：即使带图片的分支还需要 `tellraw`，它的文本前缀也不再只是随手拼字符串，而是按组件化思路整理为更结构化的 JSON 片段。
- 📌 对审核视角来说，这意味着项目不是停留在“能用就行”的脚本层，而是在 **能保持兼容的前提下，逐步把适合交给 MCDR API 的部分迁移到 MCDR API**。

### 📦 功能与运行层改进
- 🌐 `WebSocketHandler` 的初始化参数被显式化，`host`、`port` 改为从配置注入，便于部署环境调整。
- 🔁 WebSocket 启停流程更清晰，开始记录事件循环并使用更稳妥的关闭方式。
- 🖼️ 图片能力拆分为缓存、消息替换、渲染等子模块，后续继续维护图片链路时更容易定位问题。
- 🧰 `ImageHandler` 现在同时负责白名单校验反馈、加载提示、GIF 首帧提示、错误提示等用户可见交互，整体体验更完整。

### 📝 Docs
- 🇨🇳 将 `README.zh-cn.md` 替换为主文档 `README.md`，中文说明成为默认入口。
- 🇺🇸 新增 `README.en-us.md`，并与中文版本保持结构、语义和行数一致。
- 🔁 增加中英文 README 互跳入口，双语维护体验更好。
- 🖼️ 在功能说明部分补充流程预览图，帮助读者快速理解群服互通链路。
- 🔗 为 `Koishi`、`Nonebot2`、`Astrbot`、`WebSocket` 等关键词补充跳转链接。

- 📚 新增多份开发文档，尤其包括专门讨论 `RText` 使用规范和目录结构规范化的设计稿，这对后续维护和审核理解都很有帮助。
- 📦 同步更新 `mcdreforged.plugin.json` 描述文本，使插件元数据与 README 表述统一。

### 💡 这一版的意义
- 👨‍💻 对开发者：可维护性、可扩展性、可读性明显提升。
- 👥 对使用者：文档更完整，接入不同 Bot 框架和理解图片能力都更直观。
- ✅ 对审核视角：能更清楚地看到插件不仅在“加功能”，也在主动采用 `server.rtr`、`register_help_message`、`RTextList`、语言资源等 **MCDR 风格改进**。
